Architectural Decisions
DECISION 001 — Database: Supabase PostgreSQL (OVERRIDES ORIGINAL)
Database is Supabase (PostgreSQL) — cloud-hosted, RLS enabled, user_id on every table. Live from day one. There is no SQLite. Any reference to SQLite in older documents is superseded by this decision, made before S01 and permanent. Google Sheets is a read-only view layer synced on demand — never the data source.
DECISION 002 — Inventory Costing Method: FIFO
FIFO (First In First Out) as default. All four methods supported in the cost basis engine: FIFO, LIFO, Average Cost, Specific Identification. Stored in users.costing_method, locks per tax year — do not change after data accumulates without CPA guidance. Confirm FIFO selection with CPA before year-end. Critical: connects purchase tax to sale for recovery reporting.
DECISION 003 — Email Input Simplified
Order confirmation emails are ignored for data entry. User manually inputs order number, gift card last 4, and rewards rate at time of purchase. Invoice PDF is parsed after shipment — contains set numbers, prices, GWP flags. Shipping confirmation parsed for tracking number only. Email order-confirmation agents (per-retailer) are planned for future sessions — they will enrich existing orders, never create duplicate line items.
DECISION 004 — Keepa Intelligence is Build 2
Keepa has no exportable data format. API dependency is a single point of failure. Keepa intelligence layer is a standalone Build 2 — premium community add-on separate from OS core. OS functions fully without it. Aligns with core principle: own your data, rent enrichment.
DECISION 005 — Sales Import via CSV/Excel
All platform sales imports (Walmart, Amazon, Bricklink, eBay) use CSV/Excel file exports, not APIs. APIs change terms, restrict access, deprecate endpoints. File exports are stable, user-controlled, and available on every platform today. Walmart reconciliation structure confirmed: net proceeds = Product Price + Walmart Funded Savings − 8% commission − WFS fulfillment fee.
DECISION 006 — Dedicated ResellOS Gmail
A separate Gmail account is used for ResellOS. Personal Gmail forwards LEGO invoices and shipping confirmations via filter rules. ResellOS Gmail connects to MCP — personal Gmail does not. Correct onboarding pattern for all community members.
DECISION 007 — Tax Recovery at Launch
Sales tax recovery tracking is a launch feature. Value proposition: recover $5,000+ per year. Washington state first, state-configurable at onboarding. Tax recovery flows as a P&L credit after filing — it does not reopen locked cost basis records. Competing services charge 25% of recovered funds — ResellOS automates the documentation.
DECISION 008 — GWP and In-Store Flags on Line Items
GWP can occur in any channel — online or in-store. Tax exemption is determined by retailer certificate acceptance, not channel. Both are flags on the line item. GWP Philosophy C selected: proceeds_reduce_order — GWP carries $0 cost basis at receipt, net proceeds reduce originating order economic cost, reallocated across paid items only. 12-month provisional window. Configurable per user.
DECISION 009 — Tax Treatment User-Selected at Onboarding
How GWP proceeds, rewards, and cashback are treated for tax purposes is user-selected at onboarding based on CPA guidance. Cashback layer in cost basis engine is toggleable — pending CPA guidance (June 10 meeting). System applies chosen treatment consistently across all calculations.
DECISION 010 — Build Environment
VS Code + Claude Code extension. Version control: GitHub (theroyalcrate/ResellOS). Runtime: Python 3.14 with virtual environment. MCP connectors: Supabase ✓, GitHub ✗ (troubleshoot pending), Gmail + Drive (S09). Chat-Claude reaches Supabase but NOT local repo — route local file work through Claude Code.
DECISION 011 — Cost Basis Locking
Cost basis locks at settlement. Returns create P&L adjustment entries — they never reopen locked cost basis. Negative cost basis is valid and correct output — never suppressed. Common in Q4 GWP-heavy scenarios where proceeds exceed remaining set costs. Cost basis state: estimated → provisional → settled.
DECISION 012 — Intent vs Channel Fields (S8.5)
Two separate single-purpose fields. buy_reason = INTENT (why you pulled the trigger): planned | opportunistic | promo_expiration. purchase_trigger = CHANNEL (how you found the deal): community_alert | deal_software_alert | self_discovered. Both nullable — default null. Hidden from basic users, manual override for upgraded users. Email agents never set these fields — they never guess intent or channel.
DECISION 013 — Multi-Vertical Product Identity
One product_catalog table handles all verticals — not separate tables per category. A vertical field (lego | pokemon | wholesale | book | general) plus identifier_type (set_number | sku | upc | asin | isbn) normalizes product identity across categories. LEGO-specific fields (set_number, article_number) remain on line_items as legacy and continue to work. Catalog edits never retroactively affect locked cost basis records.
DECISION 014 — Order-First Onboarding
New users are not required to pre-load a catalog before entering their first order. When a line item references a product not yet in the catalog, a minimal catalog entry is created automatically — enough to track the transaction. Catalog depth grows through use. Full catalog import (CSV, Bricktap, manual) is available immediately but never required. This removes all barriers to first-session value.
DECISION 015 — Discontinuation Date Generalizes is_retiring
The LEGO-specific is_retiring concept generalizes to discontinuation_date on product_catalog — applicable to any vertical (LEGO retirement, Pokemon set rotation, wholesale EOL, book out-of-print). is_retiring remains on line_items as a LEGO-specific operational flag that defaults TRUE and drives buy decisions. The catalog field is the authoritative date; the line item flag is the fast-lookup shortcut.
DECISION 016 — Hit List as Status Layer on Product Catalog
The hit list is not a separate system — it is a status layer on top of product_catalog. A product moves: catalog entry → hit list (active) → purchased → inventory → sold. Hit list status values: active | purchased | abandoned. Abandoned preserves history — never deletes. Notes field on hit_list becomes a sourcing decision journal that feeds the intelligence layer. Purchase Planner (Phase 2) consumes hit list as its input queue.
DECISION 017
Order Edit Lifecycle & Cost Basis Trigger Gate
2026-06-01

Orders move through a defined lifecycle before cost basis is ever calculated. Cost basis is gated behind explicit user confirmation — it never runs automatically on a stub or partially-complete order.

Order status values: stub (email agent created, payment details incomplete — cost basis blocked) → pending_review (user opened, something still needs attention — cost basis blocked) → confirmed (user explicitly confirmed all inputs — cost basis calculates once) → placed (pickup orders only, awaiting pickup — cost basis blocked until pickup confirmed) → settled (12-month window passed, cost basis locked permanently).

Multiple entry paths — all valid: (1) Email agent creates stub → user confirms and adds details. (2) Manual entry at purchase → email arrives later to enrich. (3) Purchase Planner → order placed → email confirms. All three paths produce the same settled order record. The status gate applies regardless of entry path.

What email agents fill in: order number, retailer, date, line items, GWP flags, order total, tax, rewards earned, credit card last 4 if visible. Agents never fill in gift_card_last4, buy_reason, purchase_trigger, or cashback_rate.

Editable fields by status: stub/pending_review — any field editable. confirmed — only fields that do not affect cost basis (buy_reason, notes, storage_location, purchase_trigger). Fields affecting cost basis require explicit reopen to pending_review, which clears calculated cost basis and recalculates on re-confirm. settled — nothing reopens cost basis; P&L adjustment entries only.

Gift card ledger atomic write: When order moves to confirmed, two writes happen atomically — order status confirmed + cost basis calculated, and gift card balance reduced by amount applied. Reopening to pending_review reverses the prior balance reduction before applying the new one. Gift card ledger stores order_id on every debit so reversals are unambiguous.

Full Business Lifecycle

Every vertical flows through the same pipeline. LEGO, Pokemon, wholesale — same cost basis engine, same inventory tracking, same P&L reporting. The product catalog is the entry point that makes all of them work consistently.

Layer 1 — Discovery
Product Catalog
Everything that exists or could exist. Populated by import, manual entry, Bricktap upload, or auto-created on first order. Multi-vertical. One table.
↓
Layer 2 — Targeting
Hit List
Products you've decided to buy at a target price/discount. Priority levels, notes, history preserved on abandonment. Input queue for Purchase Planner (Phase 2).
↓
Layer 3 — Acquisition (Built S01–S08)
Orders → Shipments → Line Items
Manual entry via Agent 02. Invoice parsed by Agent 1A. Gift cards (Agent 05), retailer rewards (Agent 02/06), cashback (Agent 07), GWP all tracked. Cost basis engine (Agent 08) calculates true cost across all 5 layers.
↓
Layer 4 — Storage
Inventory
Unit-level tracking. One row per physical unit. FIFO date, location status, true_cost_basis per unit. WFS aging alerts. Carrying cost separate from cost basis.
↓
Layer 5 — Revenue
Sales
CSV import from Walmart, Amazon, Bricklink, eBay. Net proceeds matched to inventory units via FIFO. P&L realized. Tax recovery eligibility triggered.
↓
Layer 6 — Recovery
Tax Recovery
WA state first. Tax paid at purchase linked to sold unit. Quarterly report generated. Recovery flows as P&L credit — never reopens cost basis.
↓
Layer 7 — Intelligence (Phase 2–3)
Reporting + Signals
ROI by set, retailer, platform, buy_reason. Stamp expiry alerts. Cashback payout timing. Seasonal sourcing signals. Requires 6–12 months of tagged data to be meaningful.

Onboarding Design

Order-first design. No catalog pre-load required. Three questions, then they're in. First-session value is immediate — enter one order, see cost basis calculated.

Step 1 — What are you selling?
  • Select vertical(s): LEGO, Pokémon, Wholesale, General OA
  • Single selection is common case — UI shows only that vertical's fields
  • vertical_config loads labels and defaults for their selection
  • Multi-vertical users see all relevant options
Step 2 — Where do you source?
  • Pre-populated retailer list based on vertical
  • LEGO: LEGO.com, Barnes, Kohl's, Target, Walmart, Macy's, Best Buy
  • Other verticals: appropriate retailer defaults
  • Check what applies, skip the rest
Step 3 — Financial setup
  • State (WA first — drives tax recovery)
  • Costing method (FIFO recommended, explained simply)
  • Tax treatment (conservative / cost reduction / hybrid)
  • Gift cards: yes/no
  • Cashback platforms: which ones
  • Reseller certificate: number + expiry + which retailers accept
Catalog — No Pre-Load Required
  • Order-first: enter an order, catalog entry auto-created in background
  • Import available immediately: manual, CSV, Bricktap
  • Catalog fills in depth over time through normal use
  • No barrier between signup and first cost basis calculation
SINGLE-VERTICAL USER EXPERIENCE
A LEGO-only seller never sees Pokemon fields. A Pokemon-only seller never sees LEGO retirement logic. A wholesale seller never sees GWP tracking. The database underneath is identical — the vertical_config layer makes each user's experience feel purpose-built for them.
