Architectural Decisions
DECISION 001 — Database First
Google Sheets is the view layer, not the data layer. All truth lives in SQLite locally, upgrading to PostgreSQL at community launch. Sheets is a read-only window synced on demand. Reason: formulas break at scale, no audit trail, multi-user impossible in Sheets.
DECISION 002 — Inventory Costing Method
FIFO (First In First Out) as default for all users. Specific Identification available as a premium feature. FIFO is IRS-accepted, auditable, and defensible. Critical for connecting purchase tax to sale for recovery reporting. Confirm with CPA before year-end.
DECISION 003 — Email Input Simplified
Order confirmation emails are ignored. User manually inputs order number, gift card last 4, and rewards rate at time of purchase. Invoice PDF (from no-reply-billing03@lego.com) is parsed after shipment — contains set numbers, prices, GWP flags. Shipping confirmation parsed for tracking number only. Reduces complexity, increases reliability.
DECISION 004 — Keepa Intelligence is Build 2
Keepa has no exportable data format. API dependency is a single point of failure. Keepa intelligence layer is a standalone Build 2 — a premium community add-on separate from the OS core. OS functions fully without it.
DECISION 005 — Sales Import via CSV/Excel
All platform sales imports (Walmart, Amazon, Bricklink, eBay) use CSV/Excel file exports, not APIs. APIs change terms, restrict access, deprecate endpoints. File exports are stable, user-controlled, and available on every platform today.
DECISION 006 — Dedicated ResellOS Gmail
A separate Gmail account is created for ResellOS. Personal Gmail forwards LEGO invoices and shipping confirmations via filter rules. ResellOS Gmail is what connects to MCP — personal Gmail never touches the AI ecosystem. Correct pattern for community members to follow at onboarding.
DECISION 007 — Tax Recovery at Launch
Sales tax recovery tracking is a launch feature, not v2. Value proposition: resellers spend $400-600/year on ResellOS and recover $5,000+ in sales tax. Washington state first. State-configurable at onboarding for other states. Companies currently charge 25% of recovered funds for this service — ResellOS automates the documentation and makes filing a one-click report.
DECISION 008 — GWP and In-Store Flags on Line Items
GWP can occur in any channel — online or in-store. Tax exemption is determined by retailer certificate acceptance, not channel. Both are flags on the line item, not assumptions from purchase channel. In-store receipt parsing (OCR) is Phase 2.
DECISION 009 — Tax Treatment User-Selected at Onboarding
How GWP proceeds, rewards, and cashback are treated for tax purposes is user-selected at onboarding based on CPA guidance. Options: conservative (income when received), cost reduction (reduces cost basis), or hybrid. System applies chosen treatment consistently. CPA conversation required before year-end.
DECISION 010 — Build Environment
Primary: VS Code + Claude Code extension. Version control: GitHub. Runtime: Python with virtual environment per project. MCP connectors: filesystem → GitHub → Google Sheets → Gmail in priority order. Reason: single window, autonomous agent loop, eliminates context-switching friction.

Agent Map
01A
LEGO Online Invoice Parser
Reads LEGO invoice PDF from no-reply-billing03@lego.com. Extracts all fields. Writes to orders and line_items tables.
Input Built ✓ Phase 1
  • Handles: Standard orders, GWP (Free flag), discount column, insider points redeemed, gift card and credit card payment, split shipments
  • Output: Structured order record matched to manually entered order by order number
  • Edge cases handled: No GWP orders, quantity > 1, multiple payment methods
  • Next step: Connect output to SQLite instead of console
01B
In-Store Receipt Scanner
OCR reader for photographed or emailed in-store receipts. Handles Walmart, LEGO store, Barnes & Noble receipt formats.
Input Phase 2
  • Input: Photo upload or emailed receipt forwarded to ResellOS Gmail
  • Note: GWP can occur in-store. Tax exempt depends on retailer certificate acceptance, not channel.
  • Output: Same order/line_item structure as Agent 1A
02
Manual Order Entry
User interface for entering order at time of purchase. Captures order number, gift card last 4, rewards rate. Creates pending order record awaiting invoice.
Input Phase 1
  • Fields: Order number, retailer, gift card last 4, rewards rate (5%/2X/4X), Rakuten rate, order date
  • Status set to: pending_invoice until Agent 1A matches and populates line items
  • Rewards rate: Defaults to 5%, user adjusts for promotional events
03
Shipping Confirmation Parser
Reads shipping confirmation email. Extracts tracking number and carrier only. Updates existing order record. Supports multiple tracking numbers per order for split shipments.
Input Phase 1
  • Matches to order by: Order number in email subject or body
  • Adds to order: Tracking number, carrier, estimated arrival
  • Split shipments: Multiple tracking numbers stored per order
04
Cost Basis Engine
Calculates true cost basis per unit applying all four discount layers. Runs when invoice is matched and when GWP/rewards/cashback events complete.
Calculation Phase 1
  • Layer 1: Gift card discount % applied to order total
  • Layer 2: Retailer rewards allocated when spent from pool
  • Layer 3: Rakuten cashback allocated when received from pool
  • Layer 4: GWP net proceeds allocated proportionally to paid sets when sold
  • Output: true_cost_basis per unit in inventory table
05
Tax Allocation Engine
Allocates order-level tax proportionally to each paid line item. Flags items for tax recovery eligibility based on retailer certificate status.
Calculation Phase 1
  • Allocation method: Line item share of subtotal × order tax
  • GWP items: Zero tax allocated (no cost basis)
  • Recovery eligible: Tax paid + retailer cert on file = eligible
06
Rewards Pool Manager
Manages earn and spend cycle for each retailer's rewards pool. Variable rates per order. Tracks pool balance as a real asset.
Calculation Phase 1
  • Separate pools: LEGO Insider, Kohl's Cash, B&N Rewards, Macy's Star
  • Earn: Order pretax subtotal × order rewards rate → pool balance
  • Spend: Drawn from pool, allocated proportionally to receiving order's sets
  • Insider Points Redeem: Auto-detected from invoice, recorded as spend event
07
Cashback Pool Manager
Tracks Rakuten and external cashback. Quarterly payout cycles. Allocates received cashback back to originating orders.
Calculation Phase 1
  • Rate range: 2.5% to 15% — set per order by user
  • Payout tracking: Expected quarter vs actual receipt date
  • On receipt: Allocates back to orders that earned it, updates cost basis
08
GWP Allocation Engine
Tracks free sets from purchase to sale. Allocates net proceeds proportionally to paid sets on the originating order when GWP sells.
Calculation Phase 1
  • GWP identification: Price = Free/0.00 on invoice — auto-flagged by parser
  • Proceeds input: User enters actual net after fees and shipping when GWP sells
  • Allocation: Net proceeds ÷ paid units on order = reduction per paid unit
09
Sales Importer
Reads CSV/Excel exports from Walmart, Amazon, Bricklink, eBay. Matches sales to inventory units. Calculates true profit per sale.
Input Phase 1
  • Method: CSV/Excel file upload — no API dependency
  • Matching: Sale matched to inventory unit by set number + FIFO purchase date
  • Output: sales table populated, inventory unit marked sold, tax recovery eligibility triggered
10
Tax Recovery Report Generator
Quarterly report of all sold units with recoverable tax paid. Formatted for Washington state filing. State-configurable at onboarding.
Output Phase 1 — Launch Feature
  • Trigger: End of quarter or user request
  • Contents: Purchase date, retailer, set, tax paid, sale date, platform per unit
  • Value: Recovers $5,000+ annually for WA state resellers — headline feature
  • Status tracking: eligible → filed → reimbursed per unit
11
Google Sheets Sync
Reads from SQLite, writes human-readable views to Google Sheets. On-demand sync. Sheets is the window, not the truth.
Output Phase 1
  • Sheets tabs populated: Dashboard, Orders, Inventory, Sales, GWP Tracker, Reports, Performance
  • Sheets is read-only: No editing in Sheets — all data entry through OS
12
Performance Dashboard
Aggregated intelligence layer. ROI by set, retailer, platform. Best performing GWPs. Capital deployment recommendations.
Output Phase 2
13
Keepa Intelligence Layer
Standalone Build 2. Watchlist scanning, demand/supply signals, buy grader integrated with cost basis engine. Premium community add-on.
Build 2 Premium
  • Dependency: Keepa API — single point of failure risk noted
  • Separate from OS core: OS functions fully without this agent
  • Community pricing: Lower cost for OS members vs standalone

Specific Identification Engine
Premium upgrade to FIFO. Tracks individual units by source. Enables precise tax recovery when units from different retailers are mixed in inventory.

Tax Recovery System
// The Value Proposition
  • ResellOS subscription cost: $400-600/year
  • Average WA state tax recovery for active resellers: $5,000+/year
  • Current alternative: tax recovery companies charge 25% of recovered funds
  • ResellOS automates documentation, generates ready-to-file quarterly report, user files themselves or hands to CPA
  • This is the headline feature at community launch — not buried in the feature list
// Recovery Workflow
  • At purchase: Tax amount captured from invoice. Retailer profile checked — does cert apply? If tax was charged despite cert eligibility → flagged as potential recovery candidate.
  • FIFO matching: When a set sells, system identifies which specific purchase unit sold (first purchased = first sold). That unit's tax paid is the recovery amount. Critical when same set purchased from multiple retailers at different tax rates.
  • At sale: Unit moves to sold. Tax recovery eligibility confirmed. Record created in tax_recovery table with pending status.
  • At quarter end: Agent reads all eligible sold units for the period. Generates state-formatted report. User files. Items marked claimed.
  • At reimbursement: User records receipt. Items marked reimbursed. Amount recorded.
// Why FIFO Matters for Tax Recovery
  • Bought 50 units from LEGO.com — tax paid: $5.36/unit
  • Bought 20 units from Walmart in-store — tax exempt: $0
  • Bought 30 units from Barnes & Noble — tax paid: $3.21/unit
  • Sold 50 units — FIFO says these were the LEGO.com units
  • Tax recovery claim: 50 × $5.36 = $268.00 — clean, auditable, defensible
  • Without FIFO tracking: ambiguous which units sold, recovery amount uncertain, audit risk
Onboarding Flow
// Required at First Login — Configures All Agent Behavior
01 — State of Operation
Determines tax recovery rules applied. Washington state first. State-specific rules configured in retailer profiles. Affects tax_recovery table logic and quarterly report format.
02 — Inventory Costing Method
FIFO (default, recommended) or Specific Identification (premium). Confirm with CPA. Affects how sales are matched to purchases for tax recovery and P&L reporting.
03 — Tax Treatment Preference
How are GWP proceeds, retailer rewards, and cashback treated for tax purposes? Conservative (income when received) / Cost Reduction (reduces cost basis) / Hybrid. Must align with CPA guidance. Applies consistently to all calculations.
04 — Reseller Certificate
Certificate number, issuing state, expiry date. Which retailers accept it. Drives tax_exempt flag on line items and tax recovery eligibility. Expiry triggers renewal reminder.
05 — Retailer Default Rates
Default rewards rate per retailer (overridable per order for promotional events). Default Rakuten rate (overridable per order). Gift card discount % default per retailer.
06 — ResellOS Gmail Connection
Connect dedicated ResellOS Gmail account (not personal Gmail). Authorize Gmail MCP for invoice and shipping confirmation reading. Set up forwarding filters in personal Gmail.
07 — Google Sheets Connection
Create or connect ResellOS Google Sheet in dedicated Gmail account. Agent 11 (Sheets Sync) configures tab structure automatically on first run.
